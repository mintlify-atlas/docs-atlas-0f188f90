The Neutronium compiler provides extensive debugging capabilities to inspect the output of each compilation stage. Use these options to understand how your code is processed and to diagnose compilation issues.

## Quick Debug Mode

The `--debug` flag enables all available debug logs at once:

```bash
./compile.sh program.nt --debug
```

**Source:** `Cli.cpp:66-68`

```cpp
if (result.count("debug")) {
    opts.logCode_ = opts.logAst_ = opts.logAssembly_ = true;
}
```

This is equivalent to:

```bash
./compile.sh program.nt --log-code --log-ast --log-assembly
```

<Note>
  The `--debug` flag does NOT enable token logging. Use `--log-tokens` with `--only-lex` for that.
</Note>

## Individual Debug Flags

For more granular control, use individual logging flags to inspect specific compilation stages.

### Source Code Logging

**Flag:** `--log-code`

Outputs the raw source file contents before compilation begins.

```bash
./compile.sh program.nt --log-code
```

**Output example:**
```
fn main() -> i32 {
    return 42;
}
```

**Source:** `main.cpp:101`

```cpp
if (opts.logCode_) std::cout << fileContents << '\n';
```

**Use when:**
- Verifying the correct file is being loaded
- Checking file encoding or special characters
- Debugging file loading issues

### Token Logging

**Flag:** `--log-tokens`

Outputs the stream of tokens generated by the lexer.

```bash
./compile.sh program.nt --log-tokens --only-lex
```

<Note>
  Token logging requires the `--only-lex` flag. The compiler will warn you if you forget this:
  
  ```
  Warning: Token logging requires the --only-lex option
  ```
</Note>

**Output example:**
```
FN
IDENTIFIER
LPAREN
RPAREN
ARROW
I32
LBRACE
RETURN
INT_LITERAL
SEMICOLON
RBRACE
EOF_
```

**Source:** `main.cpp:109-112` and `Cli.cpp:105-107`

```cpp
if (opts.logTokens_) std::print("{}\n", tokenKindToString(token.kind()));
```

**Use when:**
- Debugging lexical analysis issues
- Understanding how your code is tokenized
- Investigating keyword recognition problems

### AST Logging

**Flag:** `--log-ast`

Outputs the Abstract Syntax Tree structure after parsing.

```bash
./compile.sh program.nt --log-ast
```

**Output example:**
```
CompilationUnit
  FunctionDecl: main
    ReturnType: i32
    Parameters: []
    Body:
      ReturnStmt
        IntLiteral: 42
```

**Source:** `main.cpp:129`

```cpp
if (opts.logAst_) AST::logAst(*ast);
```

**Use when:**
- Debugging parser behavior
- Understanding program structure representation
- Investigating precedence or associativity issues
- Verifying correct AST construction

### Assembly Logging

**Flag:** `--log-assembly`

Outputs the generated x86-64 assembly code.

```bash
./compile.sh program.nt --log-assembly
```

**Output example:**
```asm
.section .text
.globl main
main:
    pushq %rbp
    movq %rsp, %rbp
    movl $42, %eax
    popq %rbp
    ret
```

**Source:** `main.cpp:148`

```cpp
if (opts.logAssembly_) std::cout << assembly.str();
```

**Use when:**
- Debugging code generation
- Optimizing performance
- Understanding calling conventions
- Learning x86-64 assembly
- Investigating runtime behavior

## Flexible Log Selection

**Flag:** `--log <comma-separated-values>`

Enable multiple specific logs using a single flag:

```bash
# Enable code and AST logging
./compile.sh program.nt --log code,ast

# Enable all three
./compile.sh program.nt --log code,ast,assembly

# Enable assembly only
./compile.sh program.nt --log assembly
```

**Accepted values:** `code`, `ast`, `assembly`

**Source:** `Cli.cpp:70-86`

```cpp
const auto logs = result["log"].as<std::vector<std::string>>();
for (const auto& item : logs) {
    if (item == "code")
        opts.logCode_ = true;
    else if (item == "ast")
        opts.logAst_ = true;
    else if (item == "assembly")
        opts.logAssembly_ = true;
}
```

<Note>
  Invalid log types will cause the compiler to exit with an error:
  
  ```
  Error: Unknown log type: 'tokens'
  ```
</Note>

## Combining Debug Options

Debug flags can be combined with pipeline control flags for targeted debugging:

<CodeGroup>
```bash Debug Lexing
# See tokens during lexical analysis
./compile.sh program.nt --only-lex --log-tokens
```

```bash Debug Parsing
# See source and AST
./compile.sh program.nt --only-parse --log code,ast
```

```bash Debug Semantic Analysis
# See AST before and after semantic analysis
./compile.sh program.nt --only-sema --log-ast
```

```bash Debug Code Generation
# See AST and generated assembly
./compile.sh program.nt --only-codegen --log ast,assembly
```

```bash Full Debug
# Enable everything
./compile.sh program.nt --debug --log-tokens --only-lex
```
</CodeGroup>

## Debug Workflow Examples

### Investigating Compilation Errors

<Steps>
  <Step title="Check source loading">
    Verify the correct file is being read
    
    ```bash
    ./compile.sh program.nt --log-code
    ```
  </Step>
  
  <Step title="Check tokenization">
    Ensure lexer produces expected tokens
    
    ```bash
    ./compile.sh program.nt --only-lex --log-tokens
    ```
  </Step>
  
  <Step title="Check AST structure">
    Verify parser constructs correct tree
    
    ```bash
    ./compile.sh program.nt --only-parse --log-ast
    ```
  </Step>
  
  <Step title="Check semantic validation">
    Look for type errors or semantic issues
    
    ```bash
    ./compile.sh program.nt --only-sema
    ```
  </Step>
  
  <Step title="Check code generation">
    Inspect generated assembly
    
    ```bash
    ./compile.sh program.nt --only-codegen --log-assembly
    ```
  </Step>
</Steps>

### Understanding Generated Code

To see how your Neutronium code translates to assembly:

```bash
./compile.sh program.nt --log code,assembly
```

This shows both your source code and the generated x86-64 assembly side-by-side.

### Debugging Type Errors

When you encounter type errors:

```bash
# See the AST structure to understand type annotations
./compile.sh program.nt --log-ast --only-sema
```

The AST will show how the parser interpreted type declarations, helping you identify mismatches.

## Performance Considerations

Debug logging can impact compilation performance:

- `--log-code`: Minimal impact (just prints source)
- `--log-tokens`: Moderate impact (processes all tokens)
- `--log-ast`: Moderate impact (traverses entire tree)
- `--log-assembly`: Minimal impact (assembly already generated)

For production builds, omit all debug flags for maximum performance.
